# 內科專科題庫 — 詳解生成規格（EXPLAIN）

本任務：為 `questions.json` 中已有題目（題幹、選項、答案、subject 皆已就緒）**生成 `explanation` 欄位**。
**這是純內容生成任務：絕對不可更動任何題目的 `id`、`year`、`type`、`question`、`choices`、`answer`、`subject`。只新增/填寫 `explanation` 欄位。**

---

## 核心原則

1. **寧多勿漏、寧詳勿略**：詳解目的是讓使用者「多看多學、不要遺漏」。內容越完整越好，字數不設上限。該補的機轉、鑑別、試驗、數據、易錯點都要補。
2. **醫學正確性第一**：所有內容必須符合現行國際指引與主流共識。不確定的數據寧可不寫，不可杜撰。
3. **答案忠實**：`explanation` 必須與題目既定 `answer` 一致。**絕不可因為自己的判斷而暗示題目答案有誤或改動答案**。若覺得答案有疑慮，照既定答案寫，不要自行更正。
4. **反向題務必標清楚**：題幹含「為非／不正確／何者錯誤／不妥／並非／何者不」等否定語時，選項分析要明確區分「正確敘述（✓）」與「題目要選的那個錯誤敘述／答案（✗，正確答案）」，邏輯不可顛倒。

---

## explanation 結構

`explanation` 為一個物件，含三個欄位：

```json
"explanation": {
  "summary": "...",
  "detail": "...",
  "sources": [ { "text": "..." } ]
}
```

### summary（考點命中）
- 一句 **粗體** 點出核心考點，再接 1–2 句延伸或答案理由。
- 反向題要在 summary 點明「問何者錯誤／不是」的邏輯與答案。
- 約 80–200 字。

### detail（完整詳解）— Markdown 格式
- **字數不設上限，寧多勿漏（擴充標準）**。簡單題至少 1000 字，整合題、計算題、複選題應 1500–2200 字以上。每個小節都要寫得詳盡，不要點到為止。
- **必含** `## 選項分析`：逐項 A–E（或複選逐項 1–5）分析，正確敘述標 ✓、答案標 ✗（正確答案）。錯誤選項要 **講透**：不只說錯，還要補「正確的應該是什麼、為什麼」。
- **建議含** 以下小節（依題目彈性選用，越完整越好）：
  - `## 概念架構` 或 `## 概念定義`：先建立判斷此題所需的核心框架。
  - `## 機轉` / `## 計算` / `## 分類`：機轉題講機轉、計算題列公式與算式、分類題給分類表。
  - **分類表 / 對照表**：能用 Markdown 表格整理的知識（分期、藥物對照、診斷標準、危險分層等）盡量做成表格。
  - `## 記憶口訣`：用 blockquote（`>`）給好記的口訣或重點濃縮。
  - `## 易混淆與進階補充`：補鑑別診斷、相關試驗（名稱+結論）、cut-off 數值、相近概念對比、臨床延伸、常見陷阱。**這一節是「多看多學」的關鍵，盡量豐富。**
  - **`## 重點整理表`（必含）**：在 detail 結尾放一個 Markdown 表格，把本題核心知識濃縮成「一眼複習」的整理表。形式依題目彈性：可以是「關鍵概念 → 重點」兩欄、「疾病/藥物 → 特徵對照」、「診斷標準 / 數值 cut-off 清單」、「鑑別診斷對照」等。目的是讓使用者考前掃一眼表格就能複習本題所有考點。表格至少 4 列。
- 風格：**中英並陳**（中文 + 英文原文／縮寫）；關鍵詞用粗體；數值、cut-off、試驗名稱明確標出。

### sources（參考）
- 1–3 筆，格式 `[{ "text": "..." }]`，優先現行指引、大型試驗、標準教科書。
- 例：`{ "text": "2022 ESC/ERS Pulmonary Hypertension Guidelines" }`。

---

## 內容深度要求（寧多勿漏）

每題盡量涵蓋：
- **為什麼對／為什麼錯**：每個選項的判斷理由。
- **正確版本**：錯誤選項對應的正確知識是什麼。
- **數據與標準**：相關的 cut-off、分期標準、劑量、發生率、診斷門檻。
- **關鍵試驗**：若該主題有改變臨床實務的指標性試驗（如 IABP-SHOCK II、PROSEVA、ART、INCREASE、SENSCIS 等），點出名稱與結論。
- **鑑別與易混淆**：與相近疾病／概念的對比，常見的考試陷阱。
- **記憶輔助**：口訣、框架。

---

## 嚴格禁止

- 不可更動題幹、選項、答案、id、year、type、subject。
- 不可在詳解中宣稱題目答案有誤或自行更正答案。
- 不可杜撰不存在的試驗、數據或指引。不確定就不寫該細節。
- 不可只寫「A 對 B 錯」而不解釋；每個判斷都要有理由。

---

## 流程

1. 讀取指定範圍題目（建議一次一個專科，如「114 年腸胃科」）。
2. 為每題生成 `explanation`（summary + detail + sources），寫回該題物件。
3. 跑驗證（見下）。
4. **先停下來回報**：本批完成幾題、detail 平均/最短/最長字數、有無疑難題。**等使用者確認後**再 commit / push。
5. commit 後將 `index.html` 的 `questions.json?v=N` 版本號 +1（與題庫一起 commit），避免使用者抓到舊快取。

---

## 驗證

寫入後檢查：
- `questions.json` 仍是合法 JSON、題數不變（1000 題）。
- 本批每題都有 `explanation`，且含 `summary`、`detail`、`sources` 三欄。
- `detail` 含 `## 選項分析` 段落。
- `detail` 含 `## 重點整理表` 段落，且內含 Markdown 表格（至少 4 列）。
- **其餘欄位完全未變**（id/year/type/question/choices/answer/subject）：建議先備份再 diff，確保只新增 explanation。
- 反向題的選項分析邏輯正確（答案＝題目既定 answer）。

任一項失敗 → 不要 commit，回報問題。

---

## Git 慣例

- 確認後才 commit。commit message 用英文，例：`Add explanations for 114 gastroenterology`。
- push 指令為單一不含註解的程式碼區塊。
- 不要把 `resource/` 或 PDF 加入 git。
